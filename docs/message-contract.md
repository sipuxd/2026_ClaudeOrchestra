## JSON Message Schema

```json
{
  "messageId": "msg-<uuid>",
  "threadId": "thread-<uuid>",
  "timestamp": "ISO-8601",
  "roleSource": "Worker | Supervisor | Security | Reviewer",
  "roleSourceInstance": "Worker-1 | Worker-2 | Supervisor-1 | Security-1 | Reviewer-1",
  "roleTarget": "Worker | Supervisor | Security | Reviewer",
  "roleTargetInstance": "Worker-1 | Worker-2 | Supervisor-1 | Security-1 | Reviewer-1 | null",
  "flag": "<enum per role pair>",
  "priority": "low | normal | high | critical",
  "phase": "pre-work | work | handoff | review",
  "content": "<string, max 8000 characters>",
  "references": ["msg-<uuid>", "task-<id>"],
  "requiresResponse": true,
  "status": "pending | acknowledged | resolved"
}
```

### Field Descriptions

| Field | Type | Description |
|-------|------|-------------|
| `messageId` | `string` | Unique identifier. Format: `msg-<uuidv4>`. Generated by the sending agent. |
| `threadId` | `string` | Groups related messages into a conversation thread. Format: `thread-<uuidv4>`. The initiating message creates a new threadId; responses reuse it. |
| `timestamp` | `string` | ISO-8601 timestamp. Used for ordering and timeout detection. |
| `roleSource` | `enum` | Role of the sending agent. |
| `roleSourceInstance` | `string` | Specific instance of the sender (e.g., `Worker-1`). |
| `roleTarget` | `enum` | Role of the intended recipient. |
| `roleTargetInstance` | `string \| null` | Specific instance, or `null` for role-level multicast (delivers to all instances of that role). |
| `flag` | `enum` | Scoped per role pair. Drives routing and tells the receiver what kind of message this is. See Flag Enums below. |
| `priority` | `enum` | Determines surfacing order in the dashboard. See Priority Levels below. |
| `phase` | `enum` | The workflow phase this message belongs to. Provides context for interpreting the flag. |
| `content` | `string` | The message payload. Maximum 8,000 characters (see Size Limits). |
| `references` | `string[]` | Links to related message IDs or task IDs. Enables threading and cross-referencing. |
| `requiresResponse` | `boolean` | If `true`, the system tracks whether a response has been received. Enables stuck detection and timeouts. |
| `status` | `enum` | Lifecycle: `pending` (unread) → `acknowledged` (read by recipient) → `resolved` (no further action needed). |

## Flag Validation Matrix

The message bus **must reject** messages with invalid
`roleSource` + `roleTarget` + `flag` combinations. The
following matrix defines all legal combinations. Any
combination not listed is invalid and must be rejected with
a validation error logged.

### Legal Flag Routes

| roleSource | roleTarget | Legal Flags |
|-----------|-----------|-------------|
| Supervisor | Worker | `task-assignment`, `direction-change`, `pause`, `resume`, `check-in`, `revision-request` |
| Worker | Supervisor | `task-accepted`, `progress-update`, `task-complete`, `blocked`, `needs-guidance`, `scope-concern`, `anomaly-detected` |
| Supervisor | Security | `scan-request`, `sweep-request`, `escalation-query` |
| Security | Supervisor | `clearance-report`, `handoff-clearance`, `security-alert`, `escalation-response` |
| Worker | Security | `clearance-request` |
| Security | Worker | `clearance-granted`, `clearance-denied` |
| Supervisor | Reviewer | `review-request` |
| Reviewer | Supervisor | `review-approved`, `review-revise`, `review-rejected` |
| Worker | Worker | `sync-request`, `sync-response`, `heads-up` |

### Invalid Routes (Examples)

These are examples of messages that must be rejected:

- Worker → Reviewer (no direct path exists)
- Reviewer → Worker (no direct path exists)
- Reviewer → Security (no direct path exists)
- Security → Reviewer (no direct path exists)
- Worker sending `clearance-report` (wrong role for this flag)
- Security sending `task-complete` (wrong role for this flag)
- Any role sending to itself (e.g., Supervisor → Supervisor)

### Implementation

The router's `send()` function validates against this matrix
before writing to the target inbox. Invalid messages are:
1. Rejected (not written to any inbox)
2. Logged as a validation error with full message content
3. Counted as a potential agent malfunction (may trigger
   health check escalation after repeated violations)

## Message Ordering

### Write Ordering

Messages are written to the filesystem as individual JSON files
within the target agent's inbox directory. File naming uses the
pattern:

```
{timestamp}-{messageId}.json
```

Where `timestamp` is a zero-padded millisecond-precision
ISO-8601 string (e.g., `2026-03-07T15:30:00.123Z`) and
`messageId` is the UUID. This ensures filesystem sort order
matches chronological order.

### Read Ordering

The `receive()` function **must** return messages sorted by
`timestamp` ascending (oldest first). This guarantees that
phase transition decisions process events in the correct
causal order.

If two messages have identical timestamps (unlikely but
possible with concurrent agents), secondary sort is by
`messageId` (lexicographic) for deterministic ordering.

### Processing Order

The orchestrator's `tick()` loop processes messages in
timestamp order within each agent's inbox. Cross-inbox
ordering (messages to different agents) is not guaranteed,
but this is acceptable because agents process their own
inboxes independently.

---

## Atomic Writes

All message writes to the filesystem use the **temp file +
rename** pattern to ensure atomicity:

1. Write the JSON content to a temporary file in the same
   directory: `{inbox}/.tmp-{uuid}.json`
2. Call `fs.rename()` (which is atomic on POSIX filesystems
   at the directory-entry level) to move the temp file to
   its final name: `{inbox}/{timestamp}-{messageId}.json`
3. If the process crashes between steps 1 and 2, the temp
   file is orphaned but no corrupt message appears in the
   inbox. Orphaned temp files are cleaned up on startup.

This guarantees that `receive()` never reads a partially
written message.

### Temp File Cleanup

On engine startup, scan all inbox directories for files
matching `.tmp-*`. Delete them — they are artifacts of
interrupted writes from a previous session.

## Message Size Limits

| Field | Max Size | Rationale |
|-------|----------|-----------|
| `content` | 8,000 characters | Keeps messages within a reasonable context window budget. Large payloads (code diffs, full file contents) should be written to the reports directory and referenced via `references` field. |
| `references` | 20 entries | Prevents unbounded reference chains. |
| Total message JSON | 16 KB | Hard limit enforced by the router. Rejects messages exceeding this. |

### Handling Large Payloads

When an agent needs to communicate large content (e.g., a
Security Agent's full clearance report with file-by-file
analysis), the pattern is:

1. Write the full report to `data/teams/{team-id}/reports/`
   as a separate file.
2. Reference the report file path in the message's `content`
   field.
3. Keep the message's `content` to a summary (within 8,000
   characters).

This keeps the message bus lightweight and prevents context
window bloat when agents read their inbox.

## Message Deduplication

### Detection

The router tracks all processed `messageId` values in a
deduplication set. Before writing a message to an inbox, the
router checks:

1. Has this `messageId` already been written to any inbox?
2. If yes, reject the duplicate silently (log at debug level).

### Implementation

- Maintain an in-memory `Set<string>` of all messageIds
  processed during the current engine session.
- On engine startup, rebuild the set by scanning all inbox
  and archive directories for existing messageIds.
- The set does not need to be persisted — it is reconstructed
  from the filesystem on each startup.

### Why This Matters

LLM agents may occasionally produce the same message twice
due to non-deterministic behavior, especially if the engine
retries a prompt after a timeout. Without deduplication, the
phase controller could process the same event twice, causing
incorrect state transitions (e.g., counting one Worker's
`task-complete` as both Workers completing).

## Message Lifecycle

```
pending  →  acknowledged  →  resolved
```

| Status | Meaning | Set By |
|--------|---------|--------|
| `pending` | Message written to inbox, not yet read | Sender (on creation) |
| `acknowledged` | Recipient has read the message | Recipient (via engine) |
| `resolved` | No further action needed on this message | Engine (when response received or action completed) |

### Timeout Tracking

Messages with `requiresResponse: true` and `status: pending`
are monitored for timeouts. If a response is not received
within the configured timeout window, the engine escalates.
See [State Machine — Timeouts](./state-machine.md#timeouts)
for timeout values.

The `getPending()` function returns all messages where
`requiresResponse: true` and `status != resolved`, enabling
the orchestrator to detect stuck conversations.
